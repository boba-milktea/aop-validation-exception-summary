Aspect-Oriented Programming

Separating cross-cutting concerns from business logic

Problem

Where does logging belong?

You need to log every service method — duration, arguments, errors. UserService has 8 methods, ProductService 12, OrderService 6.
  • 26 methods total across three services
  • Same logging code copy-pasted into each one
  • Change the format? Edit 26 places.
// UserService.java public User findById(Long id) { log.info("Calling findById"); long s = System.currentTimeMillis(); User u = repo.findById(id)...; long d = System.currentTimeMillis()-s; log.info("Done in "+d+"ms"); return u; }
Repeated in every method of every service. Fragile, inconsistent, painful.
Question

What goes wrong with this @Around advice?

@Around("execution(* com.example.service.*.*(..))") public Object log(ProceedingJoinPoint pjp) throws Throwable { System.out.println("Before: " + pjp.getSignature().getName()); pjp.proceed(); System.out.println("After"); return null; }

A service method returns a Product object. What does the caller receive?

@Around controls the return value. If you do not return pjp.proceed(), the real result is silently thrown away and null is returned instead.
Concept

What is a cross-cutting concern?

  • Logic that does not belong to one layer — it runs across all of them
  • Logging — record what ran, how long it took, what failed
  • Security — check authentication before any method runs
  • Transactions — wrap database operations automatically
  • Exception handling — intercept errors from any layer
⚡
The Electricity Analogy
Your business logic = cooking, cleaning, painting. Cross-cutting logic = electricity. You do not rewire electricity in every room. You plug in and it works everywhere. AOP is the electrical system of your application.
Vocabulary

AOP core terms — four to know

TermWhat it isAnalogy
AspectThe class that holds the cross-cutting logicThe electrical panel
AdviceThe method that actually runs (the logic)The circuit for a room
PointcutAn expression selecting which methods to interceptWhich sockets are connected
Join pointA specific moment where Advice can be insertedThe moment you plug in
Advice typeWhen it runs
@BeforeBefore the target method executes
@AfterAfter the method — always, even if exception is thrown
@AroundWraps the method — most powerful, controls full execution
@AfterThrowingOnly when the method throws an exception
Solution

@Around — logging all service methods at once

@Aspect @Component public class LoggingAspect { @Around("execution(* com.example.service.*.*(..))") public Object log(ProceedingJoinPoint pjp) throws Throwable { String method = pjp.getSignature().getName(); System.out.println("Calling: " + method); long start = System.currentTimeMillis(); Object result = pjp.proceed(); // run the real method long ms = System.currentTimeMillis() - start; System.out.println("Done: " + method + " (" + ms + "ms)"); return result; } }
Write once. Every service method is now logged. Zero changes to service classes.
Pointcut: execution(* com.example.service.*.*(..)) = any method, any class in service package, any parameters.
Pointcut Syntax

Reading pointcut expressions

ExpressionMatches
execution(* com.example.service.*.*(..))All methods in all service classes
execution(* com.example.service.UserService.*(..))All methods in UserService only
execution(* com.example.service.UserService.find*(..))Methods starting with find
execution(public * *(..))All public methods anywhere
execution(* com.example..*.*(..))All methods in com.example and any subpackage
The .. in params means "any parameters". In a package path it means "this package and all subpackages".
Spring Built-in AOP

Spring already uses AOP — you just add annotations

AnnotationWhat Spring does behind the scenes
@TransactionalWraps method in a DB transaction. Commits on success, rolls back on RuntimeException.
@CacheableFirst call: runs method, stores result. Repeat calls: returns from cache, skips method entirely.
@AsyncRuns method in a background thread — user does not wait.
// Without @Transactional tx.begin(); repo.debit(from, amount); repo.credit(to, amount); tx.commit(); // or rollback on error
// With @Transactional @Transactional public void transfer(...) { repo.debit(from, amount); repo.credit(to, amount); }

Validation in Spring Boot

Reject bad data before it reaches your business logic

Problem

What happens without validation?

A client sends this JSON to your API:
{"title": "", "author": " ", "year": -500, "email": "not-an-email"}
  • Empty strings saved to the database
  • Impossible year (-500) stored as valid data
  • Invalid email passes through silently
  • Downstream services crash with confusing errors
  • Hard to trace where the bad data came from
Concept

The controller is your security checkpoint

🛡
Security Guard Analogy
A bouncer checks your ID at the door — before you get in. The controller is your bouncer. Check credentials (data) at the boundary. Do not let bad data into the VIP area (service + database).
  • Controller = entry point of the system — validate here
  • Service = business logic — should receive only valid data
  • Repository = data access — should never see garbage
Validate at the boundary. Reject early. Keep the rest of the system clean.
Setup

Add one dependency to enable validation

<dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-validation</artifactId> </dependency>
This brings in Hibernate Validator — the default implementation of the Jakarta Bean Validation spec.
AnnotationApplies toWhat it checks
@NotBlankStringNot null, not empty, not only whitespace
@NotNullAny typeMust not be null
@EmailStringMust look like a valid email address
@Min(n)NumberMust be greater than or equal to n
@Size(min,max)String, CollectionLength or size within range
@Pattern(regexp)StringMust match the regex pattern
Solution

Annotate your DTO or entity fields

@Entity public class Book { @Id @GeneratedValue(strategy = GenerationType.IDENTITY) private Long id; @NotBlank(message = "Title is required") private String title; @NotBlank(message = "Author is required") private String author; @Min(value = 1000, message = "Year must be realistic") private int year; @Email(message = "Must be a valid email") @NotBlank(message = "Email is required") private String email; }
The message attribute is what the client sees in the error response — make it helpful.
Solution

Trigger validation with @Valid in the controller

@RestController @RequestMapping("/books") public class BookController { @PostMapping public Book addBook(@Valid @RequestBody Book book) { return service.saveBook(book); } }
With @Valid: Spring checks all annotations on Book before the method runs. Bad data gets a 400 Bad Request automatically.
Without @Valid: No validation happens. Bad data reaches the service. You get a confusing 500 Internal Server Error.
The annotation on the field tells Spring what to check. @Valid tells Spring to check it. You need both.
Real-world Usage

Where validation saves you in production

User registration

  • @Email — real email
  • @Size(min=8) — password
  • @NotBlank — username
  • @Pattern — phone format

Product catalog

  • @NotBlank — product name
  • @Positive — price > 0
  • @Min(1) — stock >= 1
  • @Size(max=500) — description

Order API

  • @NotNull — customer ID
  • @Future — delivery date
  • @Min(1) — quantity

Event booking

  • @Past — birth date
  • @Digits — card format
  • @Valid — nested object

@Valid vs @Validated

Choosing the right annotation for the right situation

Problem

What if the bad data is in the URL, not the body?

// Someone calls: GET /contacts/-99 @GetMapping("/{id}") public Contact getById(@PathVariable Long id) { return service.getById(id); // -99 reaches the service! }
@Valid only validates @RequestBody objects. Path variables and query params need a different approach.
Concept

@Valid vs @Validated — when to use which

SituationUseException thrown
JSON body (@RequestBody)@ValidMethodArgumentNotValidException
URL segment (@PathVariable)@Validated on classConstraintViolationException
Query param (@RequestParam)@Validated on classConstraintViolationException
// @RequestBody → use @Valid @PostMapping public Contact create( @Valid @RequestBody Contact c) { return service.create(c); }
// PathVariable → @Validated on class @Validated @RestController public class ContactController { @GetMapping("/{id}") public Contact get( @PathVariable @Min(1) Long id) {...} }
Question

Both validations look correct — but what exception does each throw?

@Validated @RestController @RequestMapping("/users") public class UserController { // Path A: empty body with @Valid @PostMapping public User create(@Valid @RequestBody User user) { return service.create(user); } // Path B: negative id with @Min(1) @GetMapping("/{id}") public User getById(@PathVariable @Min(1) Long id) { return service.getById(id); } }

A student writes one @ExceptionHandler(MethodArgumentNotValidException.class) and expects it to catch both failures. What actually happens?

Fixed

Add @Validated to the class — method params now work

@Validated // enables method-level constraint validation @RestController @RequestMapping("/users") public class UserController { @GetMapping("/{id}") public User getById(@PathVariable @Min(1) Long id) { return service.getById(id); // Now: GET /users/-5 → 400 Bad Request } @PostMapping public User create(@Valid @RequestBody User user) { return service.create(user); } }
@Validated on the class activates Spring's proxy-based validation for all method parameters in that controller.
Best Practices

@Valid vs @Validated — rules to remember

Do

  • Use @Valid on every @RequestBody parameter
  • Use @Validated on the class for PathVariable / RequestParam validation
  • Always add a custom message to each constraint
  • Use a DTO (not the entity) for incoming request data

Don't

  • Forget @Valid — you get 500 instead of 400
  • Skip validation on PUT and PATCH, only POST
  • Validate only in the service — controller is the boundary
  • Return raw Spring stack traces to clients

Global Exception Handling

Return clean, consistent error responses to API clients

Problem

What does the client see without error handling?

{ "timestamp": "2025-01-01T10:00:00.000+00:00", "status": 400, "error": "Bad Request", "message": "", <-- useless "path": "/books", "trace": "org.springframework.web.bind... MethodArgumentNotValidException at... (300 more lines of stack trace)" }
Stack traces expose internals. Empty messages tell clients nothing. Every endpoint returns a different format.
Analogy

Be the helpful customer service desk

👥
Customer service desk analogy
When something goes wrong at a hotel, you go to the front desk. They give you a clear, polite, structured answer — not a stack of internal maintenance logs. Your @RestControllerAdvice is the front desk of your API.

Bad response

  • Raw exception class name
  • Stack trace with 200 lines
  • Different format per endpoint
  • No actionable message

Good response

  • Status: 400
  • "email": "Must be a valid email"
  • "name": "Name cannot be blank"
  • Same format everywhere
Solution

@RestControllerAdvice — one class handles all errors

@RestControllerAdvice public class GlobalExceptionHandler { @ResponseStatus(HttpStatus.BAD_REQUEST) @ExceptionHandler(MethodArgumentNotValidException.class) public Map<String, String> handleValidation( MethodArgumentNotValidException ex) { Map<String, String> errors = new HashMap<>(); ex.getBindingResult().getFieldErrors().forEach(err -> errors.put(err.getField(), err.getDefaultMessage()) ); return errors; } }
Result: {"email": "Must be a valid email", "name": "Name cannot be blank"}
Solution

Handle both exception types in the same class

// handles @Valid on @RequestBody @ExceptionHandler(MethodArgumentNotValidException.class) @ResponseStatus(HttpStatus.BAD_REQUEST) public Map<String, String> handleBody( MethodArgumentNotValidException ex) { Map<String, String> errors = new HashMap<>(); ex.getBindingResult().getFieldErrors().forEach(err -> errors.put(err.getField(), err.getDefaultMessage()) ); return errors; } // handles @Validated on @PathVariable / @RequestParam @ExceptionHandler(ConstraintViolationException.class) @ResponseStatus(HttpStatus.BAD_REQUEST) public Map<String, String> handleParams( ConstraintViolationException ex) { Map<String, String> errors = new HashMap<>(); ex.getConstraintViolations().forEach(v -> errors.put(v.getPropertyPath().toString(), v.getMessage()) ); return errors; }

Summary

Three tools for clean Spring Boot APIs

AOP → controls system behavior

  • @Aspect + @Around for logging, timing, security
  • @Transactional, @Cacheable, @Async are built-in AOP
  • Write once, apply everywhere — no business logic changes

Validation → input safety

  • @Valid on @RequestBody → MethodArgumentNotValidException
  • @Validated on class + @Min/@NotBlank on params → ConstraintViolationException
  • Reject at the boundary, keep the service clean

Exception handling → output clarity

  • @RestControllerAdvice catches exceptions globally
  • @ExceptionHandler maps exceptions to clean JSON responses
  • Consistent format across every endpoint
AOP keeps behavior clean. Validation keeps data clean. Exception handling keeps responses clean.
